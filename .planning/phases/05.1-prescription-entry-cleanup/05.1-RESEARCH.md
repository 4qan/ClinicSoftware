## Summary

This phase implements four discrete changes from the quick-5 UX research: splitting drug display into search vs. selected variants, adding amber visual indicators to ComboBox for non-standard values, renaming `dosage` to `quantity` across the data model (with Dexie migration), and ensuring no blocking validation anywhere. All changes are within existing files with well-understood patterns.

The codebase is small and well-structured. The `dosage` field appears in exactly 13 files. The ComboBox component is a single file (~150 lines) that needs a minor prop addition. The Dexie migration is straightforward since `dosage`/`quantity` is not an indexed column, just a data property on `visitMedications`.

**Primary recommendation:** Execute as three sequential concerns: (1) data model rename with migration, (2) drug display split, (3) ComboBox indicator enhancement. The rename is the riskiest (touches most files + DB migration), so it goes first.

## Architecture Patterns

### Change 1: Drug Display Split

**Current:** Single `formatDrugDisplay(drug)` function at MedicationEntry.tsx:97-105, used for both dropdown items and post-selection input value.

**Target:** Two functions:
```typescript
// For dropdown items during search (unchanged behavior)
function formatDrugSearchResult(drug: Drug): string {
  const details: string[] = []
  if (drug.saltName) details.push(drug.saltName)
  if (drug.strength) details.push(drug.strength)
  if (drug.form) details.push(drug.form)
  return details.length > 0
    ? `${drug.brandName} (${details.join(' ')})`
    : drug.brandName
}

// For input value after selection (brand name only)
function formatDrugSelected(drug: Drug): string {
  return drug.brandName
}
```

**Touchpoints:** Only MedicationEntry.tsx. The `handleSelectDrug` function (line 57) calls `setDrugQuery(formatDrugDisplay(drug))`, which changes to `setDrugQuery(formatDrugSelected(drug))`. The dropdown map (line 141) changes to `formatDrugSearchResult(drug)`.

### Change 2: ComboBox Guided-with-Override Indicator

**Current ComboBox props:** `options`, `value`, `onChange`, `placeholder`, `label`, `error`, `disabled`.

**New prop:** `showCustomIndicator?: boolean` (default: false). When true AND the current value is non-empty AND does not match any option, render an amber border + subtle info text.

**Detection logic:**
```typescript
const isCustomValue = showCustomIndicator && value.trim() !== '' &&
  !options.some(opt => opt.toLowerCase() === value.toLowerCase())
```

**Visual treatment:** Replace `border-gray-200` with `border-amber-400` and optionally show a small text hint below: "Custom value" in muted amber.

**Consumers that pass `showCustomIndicator={true}`:** The three ComboBox instances in MedicationEntry for Qty, Frequency, and Duration. The Form ComboBox (for custom drugs) does NOT get the indicator.

### Change 3: Data Model Rename (dosage -> quantity)

**Dexie Migration Strategy:**

The `dosage` field is NOT indexed (not in the `.stores()` definition). It is just a data property on `VisitMedication` objects. This means:

1. The Dexie `.stores()` definition does NOT change (no schema change needed for indexes).
2. A version(3) upgrade callback renames the property on existing records.
3. The TypeScript interface and all code references change from `dosage` to `quantity`.

```typescript
this.version(3).stores({
  // Same indexes as v2 - no index changes
  patients: 'id, patientId, firstNameLower, lastNameLower, contact, createdAt',
  settings: 'key',
  recentPatients: 'id, viewedAt',
  drugs: 'id, brandNameLower, saltNameLower, isCustom',
  visits: 'id, patientId, createdAt',
  visitMedications: 'id, visitId',
}).upgrade(tx => {
  return tx.table('visitMedications').toCollection().modify(med => {
    if ('dosage' in med && !('quantity' in med)) {
      med.quantity = med.dosage
      delete med.dosage
    }
  })
})
```

### Anti-Patterns to Avoid
- **Partial rename:** Do not rename the TypeScript field without the Dexie migration. Old records with `dosage` would silently have `undefined` for `quantity`.
- **Blocking validation:** The phase explicitly requires NO blocking. Do not add `error` prop usage on ComboBox for non-standard values. The amber indicator is informational only.
- **Coupling changes:** The drug display split is completely independent from the rename. Do not mix them in the same commit/task if possible.

## Common Pitfalls

### Pitfall 1: Dexie Migration on Non-Indexed Fields
**What goes wrong:** Developer assumes they need to change `.stores()` definition for a data-only property rename.
**Why it happens:** Confusion between Dexie indexes (declared in stores) and data properties (just stored on objects).
**How to avoid:** The `dosage`/`quantity` field is NOT indexed. Only add a version(3).upgrade() callback to rename the property on existing records. Keep `.stores()` identical to v2.
**Warning signs:** Seeing `visitMedications: 'id, visitId, quantity'` in stores would be wrong (adding an unnecessary index).

### Pitfall 2: Drug Query State After Selection
**What goes wrong:** Changing the display to brand-name-only but forgetting that `handleDrugQueryChange` clears structured fields. If the user clicks back into the drug field after selection, the shorter text triggers a re-search and clears form/strength/salt.
**Why it happens:** The input is dual-purpose: search tool AND display.
**How to avoid:** The current behavior already handles this: `handleDrugQueryChange` is called on input change (user typing), which resets fields. The `handleSelectDrug` only calls `setDrugQuery()` without triggering the change handler. Just ensure the shorter display text does not accidentally re-trigger search.
**Warning signs:** Selecting a drug, then clicking away and back, causes field reset.

### Pitfall 3: Case-Sensitive Custom Value Check
**What goes wrong:** User types "twice daily" (lowercase) and gets the amber indicator even though "Twice daily" is a valid option.
**How to avoid:** Use case-insensitive comparison: `opt.toLowerCase() === value.toLowerCase()`.

### Pitfall 4: Translation Interface Rename
**What goes wrong:** Renaming `dosage` in `MedicationFormData` and `VisitMedication` but forgetting `MedicationForInstruction` interface in translations.ts.
**How to avoid:** The grep shows translations.ts:76-78 has `MedicationForInstruction { dosage: string }`. This MUST also rename to `quantity`. And `buildUrduInstruction` accesses `med.dosage` at lines 214-215.

## Code Examples

### ComboBox Enhancement (amber indicator)
```typescript
// New prop addition
interface ComboBoxProps {
  options: string[]
  value: string
  onChange: (value: string) => void
  placeholder?: string
  label?: string
  error?: string
  disabled?: boolean
  showCustomIndicator?: boolean  // NEW
}

// Inside component
const isCustomValue = showCustomIndicator && value.trim() !== '' &&
  !options.some(opt => opt.toLowerCase() === value.toLowerCase())

// Border class modification
className={`w-full px-3 py-2 text-base border rounded-lg ... ${
  error ? 'border-red-500'
  : isCustomValue ? 'border-amber-400'
  : 'border-gray-200'
}`}

// Optional hint below input (before error message)
{isCustomValue && (
  <p className="mt-1 text-xs text-amber-600">Custom value</p>
)}
```

### Usage in MedicationEntry
```typescript
<ComboBox
  options={quantityOptions}
```
