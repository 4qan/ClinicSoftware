# Phase 22.1 Plan 04: Provider/Route/Sync Wiring Summary

**One-liner:** Splits the React provider tree at boot based on `getDeploymentMode()` -- solo mounts `SoloAuthProvider` (no `SyncProvider`), networked mounts `NetworkedAuthProvider` + `SyncProvider` exactly as today; `SyncContext.useEffect` adds an internal early return for solo (defense in depth) and `ProtectedRoute` adds an always-call-hook solo bypass (W8) so doctor-only routes stay reachable for the single solo user.

## Final App.tsx Provider Tree

```
App
└── BrowserRouter
    ├── (mode === 'solo')
    │   └── SoloProviders
    │       └── SoloAuthProvider           // useSoloAuth
    │           └── ToastProvider
    │               └── AppContent          // routes
    │
    └── (mode === 'networked')
        └── NetworkedProviders
            └── NetworkedAuthProvider      // useCouchAuth
                └── SyncProvider            // pouchDb.sync starts here
                    └── ToastProvider
                        └── AppContent
```

`getDeploymentMode()` is read once inside `App()` (not at module scope) so vitest can `vi.mock('@/db/localSettings', ...)` per-test.

## Two-Gate Sync Mitigation (T-22.1-18)

| Gate | Location | Mechanism |
|------|----------|-----------|
| 1. Provider-tree omission | `src/App.tsx` `SoloProviders` | `<SyncProvider>` is never mounted in solo. The component is never instantiated, the `useEffect` never runs, `start()` is unreachable. |
| 2. Internal short-circuit | `src/sync/SyncContext.tsx` `useEffect` first statement | `if (getDeploymentMode() === 'solo') return` runs before any other effect logic. Even if a future change mounts `<SyncProvider>` in solo, replication still cannot fire. `startSync()` (manual retry) has the same guard. |

`useSyncManager.ts` and `useCouchAuth.ts` are unchanged (`git diff` returns empty), satisfying the SPEC minimal-wrap constraint.

## ProtectedRoute Always-Call-Hook Pattern (W8)

```typescript
export function ProtectedRoute({ children, allowedRoles }: Props) {
  // Hook always at top -- rules-of-hooks safe, no lint disables.
  const { isAuthenticated, role } = useAuthContext()

  if (!isAuthenticated) return <Navigate to="/" replace />        // auth check FIRST
  if (getDeploymentMode() === 'solo') return <>{children}</>       // solo bypass (D-14)
  if (allowedRoles && !role) return <Navigate to="/" replace />   // defensive null-role redirect
  if (allowedRoles && role && !allowedRoles.includes(role)) {     // role check
    return <Navigate to="/" replace />
  }
  return <>{children}</>
}
```

- `grep -c "eslint-disable" src/components/ProtectedRoute.tsx` returns `0`.
- Auth check runs BEFORE solo bypass, so a solo unauthenticated visitor is still redirected (mitigates T-22.1-16).
- Solo bypass skips ROLE check only; not the auth check.

## AuthProvider Variant Split (Rules-of-Hooks Safe)

The original `AuthProvider` is removed and replaced by two narrowly-scoped variants. Each calls exactly one hook, eliminating the conditional-hook-call risk flagged in PATTERNS.md.

```typescript
export function SoloAuthProvider({ children })       { const auth = useSoloAuth();  /* ... */ }
export function NetworkedAuthProvider({ children })  { const auth = useCouchAuth(); /* ... */ }
```

`useAuthContext` and the `'must be used within an AuthProvider'` error message are unchanged so all downstream consumers (LoginPage, ChangePassword, Sidebar, Header, ProtectedRoute, SettingsPage, NewVisitPage, EditVisitPage, ResetNursePassword, AppContent) work in both modes without modification.

`AuthContextType` stays pinned to `ReturnType<typeof useCouchAuth>` -- Plan 03's contract test (`exposes the same 9 keys as useCouchAuth`) guarantees `useSoloAuth` is structurally compatible.
