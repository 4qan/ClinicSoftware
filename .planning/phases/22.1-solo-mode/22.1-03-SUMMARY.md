# Phase 22.1 Plan 03: useSoloAuth Hook Summary

**One-liner:** Adds `useSoloAuth` — a React hook with the same 9-key return shape as `useCouchAuth` that authenticates against a PBKDF2 envelope in localStorage instead of CouchDB `/_session`, with zero fetch calls in any code path (login, restore, changePassword, logout).

## Final Return Shape (matches useCouchAuth verbatim)

| Key | Solo behavior | Networked behavior |
|-----|---------------|--------------------|
| `isAuthenticated` | `true` after successful local verify | `true` after successful `/_session` |
| `isLoading` | `true` until session-restore effect resolves | same |
| `role` | always `'doctor'` when authenticated | `'doctor'` or `'nurse'` from `userCtx.roles` |
| `username` | persisted/recovered from sessionStorage | same |
| `credentials` | always `null` (no Basic auth) | base64(`username:password`) |
| `login(u, p)` | seeds-or-verifies against PBKDF2 envelope | `fetch(/_session)` |
| `logout()` | clears state + sessionStorage + legacy key | same |
| `changePassword(c, n)` | rewrites PBKDF2 envelope after verify | `fetch(/_users)` |
| `resetNursePassword(_, _)` | typed no-op | `fetch(/_users/...:nurse)` |

The 9 keys match exactly. `AuthProvider` (Plan 04) can swap hooks without consumer changes.

## Session Payload Differences

```jsonc
// useCouchAuth writes:
{ "username": "doctor", "credentials": "ZG9jdG9yOmRvY3RvcjEyMw==", "role": "doctor" }

// useSoloAuth writes:
{ "username": "doctor", "role": "doctor" }
```

Same `SESSION_KEY = 'clinic_couch_session'` (D-19 parity). Plan 04 ensures only one provider tree mounts per page load, so the key collision is benign — JSON.parse on either side tolerates missing fields.

## First-Launch Seeding (D-06)

```typescript
const stored = getSoloCredentials()
if (!stored) {
  if (username === 'doctor' && password === 'doctor123') {
    const env = await hashPassword('doctor123', 'doctor')
    setSoloCredentials(env)
    // ... authenticate
  }
  return { ok: false, error: 'invalid_credentials' }
}
```

Verified by Test 1 (success path) and Test 2 (wrong default → no envelope written, no auth).

## Session Restore (D-19)

```typescript
const env = getSoloCredentials()
if (!env || env.username !== parsed.username) {
  sessionStorage.removeItem(SESSION_KEY)
  return  // drops stale session
}
```

Two test cases lock this:
- "restores session when both flag and envelope present" → authenticated after isLoading=false
- "drops session when envelope was wiped" → `sessionStorage.getItem(SESSION_KEY) === null` after restore

## Zero-Fetch Strategy (D-17)

Every test installs a fetch spy in `beforeEach`:

```typescript
fetchSpy = vi.fn(() => Promise.reject(new Error('fetch should not be called in solo mode')))
globalThis.fetch = fetchSpy as unknown as typeof fetch
```

And every test ends with:

```typescript
function expectNoFetchCalls() {
  const offending = fetchSpy.mock.calls.filter(([url]) => {
    const s = String(url)
    return s.includes('5984') || s.includes('_session') || s.includes('_users') || s.startsWith('http')
  })
  expect(offending).toEqual([])
}
```

URL-shape filtering means future incidental jsdom calls (none observed) won't trigger false negatives, while every CouchDB-shaped URL fails the assertion. Plan 07 adopts this same helper for the broader route/sync coverage.

## Threat Mitigations Applied

| Threat ID | Mitigation | Evidence |
|-----------|------------|----------|
| T-22.1-09 (Spoofing — known defaults) | Defaults are intentional first-launch friction; no remote attack surface; ChangePassword (Plan 06) lets the doctor rotate them | `DEFAULT_USERNAME` / `DEFAULT_PASSWORD` constants only fire when `getSoloCredentials() === null` |
| T-22.1-10 (Tampering envelope) | Envelope binds username + hash + iterations; out-of-scope per SPEC (single-machine threat model) | `verifyPassword(password, stored)` rejects when any field doesn't reproduce the hash |
| T-22.1-11 (Info disclosure session payload) | Payload omits credentials; username is single-user public anyway | Session restore code reads only `username` and `role` |
| T-22.1-12 (Spoofing on restore) | D-19: restore re-verifies envelope presence + username match | Test "drops session when envelope was wiped" |
| T-22.1-13 (Privilege elevation) | Role hardcoded to `'doctor'` in solo; networked features gated by Plan 04+ at provider boundary, not by role string | Every authenticated state object sets `role: 'doctor'` |
| T-22.1-14 (SESSION_KEY collision) | JSON.parse tolerates extra/missing fields; cross-mode reads drop session rather than crash | Restore validates `parsed.username && parsed.role === 'doctor'`; defensive try/catch on parse |

`security_block_on: high`. No HIGH-severity unmitigated threats.

## Deviations from Plan

**None — plan executed exactly as written.**

Two minor implementation hardenings that are not deviations from intent but worth noting for future readers:

1. The session-restore `JSON.parse` is wrapped in its own try/catch (in addition to the outer try/finally that owns `setIsLoading(false)`). The plan's example code only had the outer try/finally; corrupted JSON would have thrown. This is consistent with Plan 01's `getSoloCredentials` defensive parse pattern (T-22.1-03 family) and was the safer default. Behaviorally identical to spec since corrupted session payloads should drop the session.

2. The `expectNoFetchCalls` helper filters by URL shape (5984, _session, _users, http*) rather than asserting `mock.calls.length === 0`. The plan's narrative text says "no call URL contains the CouchDB host" (D-17), so URL-shape filtering aligns with intent and avoids future false positives if any non-CouchDB fetch crops up in jsdom internals. No such fetches are made today.
