### MEDIUM

#### MD-01 — Username not bound into PBKDF2 input; envelope verification is independent of stored username

**File:** `src/utils/passwordHash.ts:24-42`, `src/auth/useSoloAuth.ts:122`
**Severity:** Medium
**Confidence:** 90%

**Issue:** `hashPassword(password, username)` stores `username` in the envelope as a metadata field but does NOT mix it into PBKDF2 input. Only `password` is fed to `crypto.subtle.deriveBits`; `username` is preserved verbatim in the returned envelope. Verification re-derives bits from `password + envelope.salt + envelope.iterations` only, then compares to `envelope.hash`. The `username !== stored.username` check in `useSoloAuth.login` (line 122) lives at the application layer, not in the cryptographic binding.

Consequence: a localStorage attacker who can rewrite `soloCredentials` could swap the `username` field on a captured envelope without invalidating the hash, then log in under the new username with the original password. The threat model explicitly accepts physical-machine compromise (CONTEXT D-15, T-22.1-10), so this is documented-as-accepted. Flagging because the JSDoc on `hashPassword` reads as if `username` is part of the hash binding (the parameter is positional and named); a reader could reasonably assume username is cryptographically bound.

**Fix (optional, for future hardening):** Either (a) include username in the salt-derivation input (e.g., `TextEncoder().encode(username + ':' + password)` for the `importKey` step), or (b) update the JSDoc on `hashPassword` to clarify that username is metadata-only and the cryptographic binding is `password + salt`. Option (b) is the cheaper, more honest fix.

```typescript
// Suggested JSDoc clarification on hashPassword:
/**
 * PBKDF2-hash a password. The username is stored in the envelope as metadata
 * for the application-layer username check in useSoloAuth.login; it is NOT
 * part of the cryptographic binding. An attacker with localStorage write
 * access can swap the username field; the threat model (single-machine PWA)
 * accepts this.
 */
```

#### MD-02 — `useSoloAuth.changePassword` reads `state.username` from a stale closure when invoked outside a re-render boundary

**File:** `src/auth/useSoloAuth.ts:150-168`
**Severity:** Medium
**Confidence:** 70%

**Issue:** `changePassword` is wrapped in `useCallback` with `[state]` as the dependency. Because `state` is the dependency, the callback is re-created on every state change, so consumers calling it via the React tree will see the latest closure. However, in the test at `useSoloAuth.test.tsx:159-177`, `changePassword` is called immediately after `login()` resolves inside `act(async)`. Because the callback was destructured from `result.current` at the top of the `act` block, the test path is fine.

The concern: the destructure pattern in components (e.g., `ChangePassword.tsx:5`: `const { changePassword } = useAuthContext()`) is re-read on every render, so it's always fresh — but if any consumer caches the function reference across renders (e.g., a `useRef` or stale closure), they'd see old `state`. No current consumer does this, so the bug is latent only.

The dependency `[state]` causes the function identity to change on every state mutation, which can cause unnecessary re-renders in any consumer using `changePassword` in a `useEffect` deps array. None do today; flagging as a minor footgun.

**Fix:** Either (a) read `username` via a ref pattern to keep `changePassword` stable, or (b) use the functional `setState` pattern and remove `state` from deps. Lowest-risk option: leave as-is and accept that consumers must always read `changePassword` from `useAuthContext()` in the same render.

```typescript
// Option A (refactor with ref):
const stateRef = useRef(state)
useEffect(() => { stateRef.current = state }, [state])
const changePassword = useCallback(async (current, next) => {
  const { username } = stateRef.current
  // ...
}, [])
```

### LOW

#### LO-01 — `verifyPassword` length-check leaks fixed-length information before constant-time compare

**File:** `src/utils/passwordHash.ts:66`
**Severity:** Low
**Confidence:** 85%

**Issue:** `if (candidate.length !== envelope.hash.length) return false` runs BEFORE the constant-time XOR-OR loop. In principle this leaks via timing whether the candidate hash length matches the stored hash length. In practice, both are derived from `KEY_LENGTH_BITS = 256` (fixed) and base64-encoded, so the length is always the same constant for any password input — the early return only fires if the stored envelope was tampered with to have a non-32-byte hash. Not a real attack surface on a single-machine PWA; flagging because the comment "Constant-time string compare (cheap, no real risk on single-machine PWA but trivially safe)" implies the entire compare is constant-time, but the length-check branch is not. Cosmetic.

**Fix:** None required. Optionally remove the length check (the XOR loop would fail on length mismatch anyway, since `candidate.charCodeAt(i)` for `i >= length` returns `NaN` and `NaN ^ x === x`, producing a non-zero diff). Or keep it and update the comment to acknowledge the early return.

#### LO-03 — `useSoloAuth.login` first-launch path lacks idempotency / double-submit guard at the hook layer

**File:** `src/auth/useSoloAuth.ts:99-117`
**Severity:** Low
**Confidence:** 70%

**Issue:** On first launch, `login('doctor', 'doctor123')` calls `await hashPassword(...)` (~100ms). If the user double-clicks the submit button before the first promise resolves, two concurrent `hashPassword` calls would each call `setSoloCredentials` and write a session. The two envelopes would have different salts, so the second overwrite sets the canonical envelope; both set the same session payload. Net effect: one envelope wins, no functional break, but two `setState` calls and two PBKDF2 derivations.

`LoginPage.handleLogin` has `isSubmitting` which prevents the second click reaching the hook, so the user-facing path is fine. The hook itself does not gate against concurrent calls. Programmatic callers (tests, future flows) could trigger this.

**Fix:** None required. If desired, add a `loginInFlightRef` ref and short-circuit when set. Lowest-priority hardening.

#### LO-04 — Solo session payload uses the same `SESSION_KEY` as networked, and `useCouchAuth` parses it as a `credentials`-bearing object without defensive guards

**File:** `src/auth/useSoloAuth.ts:18`, `src/auth/useCouchAuth.ts:55-66`
**Severity:** Low
**Confidence:** 75%

**Issue:** Both hooks use `SESSION_KEY = 'clinic_couch_session'`. The solo payload is `{ username, role }` (no `credentials`); the networked payload is `{ username, credentials, role }`. If a user toggles `deploymentMode` from solo → networked at the localStorage layer without clearing sessionStorage (manual override or migration glitch), `useCouchAuth.init` destructures the parsed object and reads `credentials` (undefined). It then calls `fetch(${url}/_session, { headers: { Authorization: Basic undefined } })`, gets 401, and clears the session. Self-healing and behaviorally correct.

The TypeScript cast `as { username: string; credentials: string; role: ... }` is a lie when the payload came from solo mode — `credentials` is undefined. No runtime error because the next operation is string interpolation (`Basic ${undefined}`), which produces `Basic undefined` and the server 401s. Defensive and self-healing, but the type cast hides the cross-mode contract drift.

**Fix:** None required for 22.1 scope. For Phase 23 / future hardening: either (a) split SESSION_KEY per mode (`clinic_solo_session` vs `clinic_networked_session`), or (b) add a runtime check in both `init` paths that the payload shape matches the expected mode.
