# Phase 22.1 Plan 06: SettingsPage Solo-Aware (Networking tab + Upgrade card + ?tab= deep-link) Summary

Settings page now branches on deployment mode: in solo, the "Networking" tab (renamed from "Sync") is rendered visibly-but-disabled with the only body content being a disabled "Add a second computer" upgrade card; ResetNursePassword is suppressed; the page reads `?tab=` from the URL on mount so Plan 07 can deep-link into the Networking tab for render assertions. In networked mode, behavior is byte-equivalent to current main, with the Phase 22 sync status JSX cleanly extracted into `src/pages/NetworkedSyncTab.tsx` so `useSyncContext` is never called when SyncProvider isn't mounted.

## What Was Built

### Task 1 — NetworkedSyncTab extraction (commit `b9b97d9`)

New file `src/pages/NetworkedSyncTab.tsx`:
- Exports `NetworkedSyncTab()` — calls `useSyncContext` and renders the Phase 22 status dot, last-synced line, error message, and retry button.
- `SYNC_STATUS_CONFIG` constant and `formatRelativeTime` helper co-located here (only used by this component).
- JSX copied verbatim from the previous SettingsPage Sync tab block. No semantic change.
- Standalone valid module: `npx tsc --noEmit` passes after this task even before SettingsPage imports it (per the plan's W9 isolation requirement).

**Why extract?** Task 1 is the regression-risk refactor; Task 2 is the new feature. Splitting them keeps the diffs reviewable and isolates the `useSyncContext` deferral, which is the actual mitigation for T-22.1-25 (calling `useSyncContext` outside `SyncProvider` would crash in solo).

### Task 2 — SettingsPage solo branches + deep-link (commit `d7d27db`)

Rewritten `src/pages/SettingsPage.tsx`:

- **Tab label flip (D-09):** `{ key: 'sync', label: isSolo ? 'Networking' : 'Sync' }`. Internal key stays `'sync'` so the existing `SettingsCategory` union and downstream conditional render checks need no edits.

- **Disabled tab button (UI-SPEC Component #2 + Accessibility Notes):**
  ```
  disabled = isSolo && tab.key === 'sync'
  - onClick: disabled ? undefined : original handler
  - aria-disabled: disabled || undefined
  - tabIndex: disabled ? -1 : undefined
  - title: disabled ? 'Networking is not available in solo mode' : undefined
  - className when disabled: 'px-4 py-2 text-sm font-medium rounded-full bg-gray-100 text-gray-600 opacity-50 cursor-not-allowed'
  ```
  Uses `onClick={disabled ? undefined : ...}` — defense-in-depth so DOM edits removing CSS cannot re-enable the click.

- **ResetNursePassword gating (D-12):** Account tab now wraps the doctor block in `{!isSolo && role === 'doctor' && ...}`. Solo has no nurse account, so the panel must not appear.

- **Upgrade card (D-13, UI-SPEC Component #3 verbatim):** Solo Networking tab body renders only:
  ```html
  <div className="bg-white border border-gray-200 rounded-lg p-4">
    <h3 className="text-lg font-bold text-gray-900 mb-1">Add a second computer</h3>
    <p className="text-sm text-gray-500 mb-4">Coming soon.</p>
    <button disabled className="bg-gray-100 text-gray-400 cursor-not-allowed px-4 py-2 text-sm rounded-lg min-h-[44px]">Coming soon</button>
  </div>
  ```
  No sync status section in solo (UI-SPEC #2 overrides D-09's preserve-and-disable hint).

- **NetworkedSyncTab swap-in:** `{activeCategory === 'sync' && !isSolo && <NetworkedSyncTab />}`. SettingsPage no longer calls `useSyncContext` at all — the call is deferred into the imported component, which only mounts in networked mode.

- **B2 — URL `?tab=` deep-link:** New `resolveInitialCategory(searchParams)` whitelist parser:
  ```
  Accepts: account | clinic | data | print | sync | networking
  Both 'sync' and 'networking' → internal key 'sync' (works in both modes)
  Invalid or missing → 'account'
  ```
  `useSearchParams` from react-router-dom; lazy `useState` initializer reads on mount only (no router sync after that — user clicks drive `setActiveCategory` as today). T-22.1-32 mitigated by the whitelist Set check.
