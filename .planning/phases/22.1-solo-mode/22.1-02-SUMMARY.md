# Phase 22.1 Plan 02: PBKDF2 Password Hash Utility Summary

**Zero-dependency PBKDF2 SHA-256 password hashing via Web Crypto, with jsdom polyfill so vitest can exercise it.**

## Accomplishments
- `hashPassword` and `verifyPassword` async functions using native `crypto.subtle` PBKDF2
- 16-byte random salt, SHA-256, 100k iteration floor (D-05)
- `verifyPassword` reads `envelope.iterations` so future iteration bumps do not invalidate stored credentials
- jsdom polyfill in `src/__tests__/setup.ts` falls back to `node:crypto` `webcrypto` when `globalThis.crypto.subtle` is missing
- 8 passing unit tests covering envelope shape, salt randomness, salt size, round-trip verify, wrong/empty password, and tampered-iterations rejection

## Decisions Made
- **Inline `SoloCredentialEnvelope`**: Plan 01 (parallel wave 1) creates `src/db/soloCredentials.ts` with the canonical type. That file is not in this worktree. Defining the type inline keeps Plan 02 self-contained, compiles cleanly, and tests pass. Both definitions are byte-identical at the type level so post-merge there is no compile failure even though `passwordHash.ts` and `soloCredentials.ts` both declare the interface. Plan 03 (wave 2, post-merge) should import from `@/db/soloCredentials` (canonical home); the duplicate in `passwordHash.ts` can stay or be replaced with a re-export.
- **Constant-time compare in verifyPassword**: XOR-OR over base64 strings. No real attack surface on a single-machine PWA, but trivially safe and the cost is one short loop.
- **`@ts-expect-error` on global crypto assignment in setup.ts**: Node's `webcrypto` types differ slightly from DOM's `Crypto`, but the surface used at runtime (`subtle`, `getRandomValues`) is identical. Annotated rather than coerced silently.

- **T-22.1-08** (jsdom missing crypto.subtle) — mitigated by Task 1 polyfill; Task 3 tests run end-to-end against the polyfilled global.
